## 3 — Module Rundowns (Detailed)

### A. Ministry & Family Hub
**Main capabilities**
- Create a Sub‑Bot per contact with: persona profile, memory bucket (interactions), language preference, and posting style.
- Prayer features: schedule recurring prayers, group prayer rooms, save answered prayers, track prayer metrics (requests vs answered).
- Events: create events, send invites, RSVP tracking, auto-reminders.
- Media: worship playlists, sermon uploads, and sermon timestamps.

**Security & privacy**
- End-to-end encryption for private prayer rooms and messages.

**Example flows**
- New member onboarding → ask for phone, role, favorite scripture → create sub-bot.
- Family emergency alert → start an instant prayer room + push alert to trusted channels.

---

### B. Crypto & Trading Suite

#### 1) Scalper Engine
- **Purpose:** Capture small intraday moves with fast in/out trades and tight risk controls.
- **Strategies:** EMA crossover (fast/slow), RSI mean-reversion, orderbook sniffing, VWAP pullbacks.
- **Safety:** paper-mode by default; simulated execution includes fee and slippage modelling.
- **Config:** timeframe, order_size, max_slippage, stop_loss_pct, take_profit_pct, risk_per_trade.
- **Outputs:** trade logs, PnL, trade heatmaps, expectancy reports.

#### 2) Slippage Guard & Sandwich Detection
- **Purpose:** Avoid money lost to front-running and sandwich attacks.
- **Mechanics:** simulate swaps against current liquidity, compute expected slippage, block if slippage > threshold.
- **Mempool Checks:** pre-check pending txs for risky sequences; throttle gas to avoid predictable timing.

#### 3) MEV Scanner
- **Purpose:** Monitor mempool for large swaps and risky patterns; optionally interact with private relayers.
- **Components:** mempool listener, heuristics engine (sandwich probability), alerting & score system.
- **Notes:** Private relayer interaction is sensitive — use only with legal counsel and clear policies.

#### 4) Sniper / Launch Monitor
- **Purpose:** Detect new token contract creations and provide pre-specified buy rules.
- **Safety:** manual confirmations, whitelist checks, max slippage, token contract audits (basic signature checks).

#### 5) Arbitrage Engine
- **Purpose:** Identify and route profitable legs across DEXs/CEXs.
- **Mechanics:** collect orderbook/ticker data, simulate round-trip with fees, route via cheapest path, consider gas vs profit.

#### 6) Signals Hub
- **Purpose:** Publish/consume trade signals, sign them, and optionally auto-execute based on rule sets.
- **Integrations:** Telegram bot, Webhooks, signed messages for authenticity.

#### 7) Telegram Integration
- Admin bot to receive signals, forward trusted ones, manage group roles, and accept operator commands (backtest/run/pause).

## 4 — Pseudocode & Samples

### Kingdom Scalper (safe default)

```pseudo
inputs: symbol, timeframe, order_size, max_slippage, stop_loss_pct
loop:
  candles = fetch_candles(symbol, timeframe)
  ema_fast = ema(candles.close, 8)
  ema_slow = ema(candles.close, 21)
  rsi = rsi_indicator(candles.close, 14)

  if ema_fast > ema_slow and rsi < 70:
    price = get_best_price(symbol)
    simulate = simulate_swap(symbol, order_size, gas_price_hint)
    if simulate.slippage <= max_slippage and ethics_check('scalp'):
      place_limit_or_ioc(price, order_size, pre_sign_prayer=True)
      set_stop_loss(price * (1 - stop_loss_pct))

  sleep(poll_interval)
```

## 5 — Backtest Plan (Scalper)

- **Data:** 1m/5m OHLCV for symbol(s) for last 180 days (or from exchange historical feeds).
- **Walk-forward:** split into training (60%) / validation (20%) / forward-test (20%).
- **Metrics:** Sharpe, Sortino, max drawdown, expectancy, trade win-rate, average hold time.
- **Sanity:** include slippage modeling, maker/taker fees, gas/tx costs.

## 6 — MEV Watcher Architecture (Detailed)

- **Components:**
  - Mempool listener (node or public mempool feeds)
  - Heuristics/Scoring engine for sandwich & back-run risks
  - Alert & Ops channel (Telegram/Slack) with payloads
  - Optional private-relayer interface (Flashbots-like) for protection
- **Data:** pending tx size, path, token liquidity, gas price, block basefee, contra-positions
- **Latency:** prioritize sub-second event stream processing; prefer local node or premium mempool feed.

**Legal caution:** always verify jurisdiction rules before using private-relayer or protective strategies.

## 7 — Telegram Signal Workflow (Detailed)

- **Flow:** Signal originator → sign (private key) → Telegram channel → bot verifies signature → simulator → operator decision or auto-exec.
- **Security:** signed signals to prevent spoofing; rate limits; allow-list of originators.
- **Auto-exec gates:** minimum simulation profit, max slippage, ethics check, and approvals if live funds involve mission accounts.
